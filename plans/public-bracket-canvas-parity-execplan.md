## Purpose / Big Picture

After this change, the public bracket widget will stop rendering simple round columns and will instead render the same bracket geometry as the schedule page. A user opening an embedded bracket will see the same horizontal spacing, vertical lane spacing, elbow connectors, and arrowheads that appear in the in-app `TournamentBracketView`, while the schedule page will keep its editing controls and continue using the same geometry engine.

The visible proof is straightforward. Open a playoff or tournament event in the schedule page and open the matching public bracket widget. The match cards do not need to be identical in every detail, but the bracket skeleton must match: the same rounds, the same spacing rhythm, and the same connector paths.

## Surprises & Discoveries

- Observation: `src/lib/bracketViewCore.ts` already centralizes division and root-match helpers, so the missing shared piece is the lane layout and connector engine, not the event-selection logic.
  Evidence: the schedule page and public widget already both import `buildBracketDivisionOptions`, `getBracketRootMatches`, and `pickPreferredBracketRootMatch`.

- Observation: `TournamentBracketView.tsx` uses fixed geometry constants, not measured DOM size, so extracting parity is feasible without a browser-only layout pass.
  Evidence: the component hardcodes `CARD_W = 288`, `CARD_H = 200`, `GAP_X = 48`, `GAP_Y = 12`, and computes all positions from match graph traversal.

- Observation: the public widget tests assert the old column renderer by looking for `.bracket-columns` and round labels.
  Evidence: `src/app/embed/[slug]/[kind]/__tests__/route.test.ts` currently checks for `Winners Bracket`, `Losers Bracket`, and the column-oriented markup generated by `renderBracketColumns`.

- Observation: `MatchCard.tsx` cannot be reused directly in the public embed because it is a client component and depends on `next/image`.
  Evidence: the file begins with `'use client'` and imports `Image` from `next/image`, so importing it into the route-rendered widget would pull client-only runtime requirements into the embed path.

- Observation: public scheduler-loaded playoff graphs may populate relation objects while leaving `previous*Id` and `winnerNextMatchId` fields as `null`.
  Evidence: the real `Test league` event loaded through `loadEventWithRelations()` produced playoff matches where bracket progression lived in `previousLeftMatch` / `winnerNextMatch`, while the parallel id fields were `null`. The shared layout engine originally treated `null` ids as an instruction to ignore the relation, which collapsed the public widget to the root match only.

## Decision Log

- Decision: keep the public embed as a route-rendered HTML document instead of converting the widget to a normal Next.js page.
  Rationale: the current route avoids the normal site shell inside iframes. The shared renderer can still be used by returning pre-rendered markup from the route.

- Decision: share the geometry engine and canvas frame first, while allowing the schedule page and public widget to use different card bodies.
  Rationale: the user asked for matching spacing and arrows. Those are defined by the layout engine and connector overlay, not by whether every line of card content is identical. This keeps the public embed lightweight and avoids pulling the schedule editor into public widgets.

- Decision: preserve the schedule page’s current winner-lane and loser-lane traversal rules exactly during extraction, including its handling of cross-bracket links and “unplaced matches”.
  Rationale: those rules already have tests and represent the current product behavior. The extraction should not silently reinterpret the bracket graph.

## Context and Orientation

The schedule-page bracket view lives in `src/app/events/[id]/schedule/components/TournamentBracketView.tsx`. It is a client component that renders a control bar, a zoomable scroll surface, a bracket canvas, and an optional dock of “unplaced matches” that are not connected to the chosen bracket tree. The actual geometry is computed in that file today. Geometry means the horizontal round positions, the vertical spacing between sibling branches, the total canvas size, and the elbow-shaped SVG connector paths.

The schedule page card body lives in `src/app/events/[id]/schedule/components/MatchCard.tsx`. That card has product-specific editing affordances and schedule metadata. The layout engine assumes a fixed card width and height that match this card.

The public bracket widget data loader lives in `src/server/publicWidgetBracket.ts`. Today it selects the right event and division but then collapses the bracket into simple `winnersColumns` and `losersColumns`. The public embed route in `src/app/embed/[slug]/[kind]/route.ts` turns those columns into stacked HTML cards. That is why the public widget does not look like the schedule page.

The already-shared root and division helpers live in `src/lib/bracketViewCore.ts`. This file resolves division keys, finds bracket-capable matches, picks a preferred root match, and gathers the connected matches for one division. That file should remain the source of truth for event and division selection. The new work should complement it with geometry helpers rather than duplicating or replacing it.

In this plan, a “lane” means one rendered branch family at a time: winners or losers. A “canvas” means the relatively positioned wrapper that contains absolute-positioned match cards plus an SVG overlay. A “geometry engine” means a pure function that receives matches and returns card positions, connector lines, and overall canvas size without touching the DOM.

## Plan of Work

Start by extracting the schedule page’s pure graph and geometry logic into a new file such as `src/lib/bracketCanvasLayout.ts`. Move the helpers that decide which matches belong to a lane, how child matches are discovered for winners and losers traversal, how rounds are numbered, how vertical centers are spaced, and how elbow connectors are drawn. The new module must be generic enough to work with both the schedule page’s front-end `Match` objects and the server-side scheduler `Match` objects used by public widgets.

Once the pure module exists, add a shared React component that only knows how to render a canvas from already-computed layout data. This component should live outside the schedule folder because both schedule and public widget code must import it. It should render a relatively positioned wrapper, one absolutely positioned card slot per match, and an SVG overlay with the same connector path shape and arrowhead marker used by the schedule page. It must not own zoom state, modal state, or widget-specific controls.

Then refactor `TournamentBracketView.tsx` to use the new layout module and canvas component. Keep the current control bar, zoom behavior, score modal behavior, and unplaced-match dock. The schedule page should continue rendering `MatchCard` and continue exposing the same testing hooks such as absolute wrappers around each card. This is a replacement of internal implementation, not a behavior rewrite.

After that, update `src/server/publicWidgetBracket.ts` so it stops returning round columns. Instead, it should return enough information to render winners and losers canvases: the selected division, the selected event, and per-lane match card view models keyed by match id together with computed layout results from the shared module. The widget route can then render the same canvas frame for each lane using a public-safe card body that shows time, field, teams, and score lines without schedule editing affordances.

Finally, update tests to follow the new renderer. The route test should assert the presence of the canvas/SVG markup and public cards instead of the old `.bracket-columns` structure. If needed, add a focused unit test for the new geometry module or extend `TournamentBracketView` tests to prove that extracted layout still places matches and connectors consistently.
