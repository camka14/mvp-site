# LEG-001 v1.6.13 Canonical API Contract

This inventory records the compatibility evidence required before BracketIQ stops emitting Appwrite-style response aliases. The supported-client floor is the mobile tag `v1.6.13`, commit `50045cc351dcd70f47603ea1c31e92fcd439762e`. “Canonical” means the server sends `id`, `createdAt`, and `updatedAt`, and represents an open-ended event with `end: null`; it does not send `$id`, `$createdAt`, or `$updatedAt`.

## Executable floor fixture

The mobile repository commit `245f6a0a88e674ac13751e6a17155b6751e40f64` adds `CanonicalOnlyContractFloorTest`. The same source was run from the exact `v1.6.13` tag and passed two tests. Its JSON contains no dollar-prefixed keys. It proves that `EventApiDto`, `TeamApiDto`, `MatchApiDto`, and `UserProfileDto` resolve canonical IDs, and that `EventApiDto` accepts `end: null` when `noFixedEndDateTime` is true.

Run the checked-in current-client fixture from `/Users/elesesy/StudioProjects/mvp-app-critical-audit`:

    ./gradlew :core:network:testDebugUnitTest --tests \
      'com.razumly.mvp.core.network.dto.CanonicalOnlyContractFloorTest'

The expected result is two passing tests. The exact-tag replay recorded in `plans/retire-pre-1-6-13-api-compatibility-execplan.md` also reports two passing tests from commit `50045cc3`.

## Resource-family source inventory

The following v1.6.13 decoders accept canonical server records. These paths are relative to the mobile repository and were inspected with `git show v1.6.13:<path>`.

- Event, team, match, and user: `core/network/src/commonMain/kotlin/com/razumly/mvp/core/network/dto/EventDtos.kt`, `TeamDtos.kt`, `MatchDtos.kt`, and `AuthDtos.kt` declare canonical `id` fields. Their conversion functions choose `id` before an optional legacy fallback. `EventApiDto.end` is nullable.
- Fields: `core/network/src/commonMain/kotlin/com/razumly/mvp/core/network/dto/FieldDtos.kt` decodes `FieldsResponseDto.fields` into `core/model/src/commonMain/kotlin/com/razumly/mvp/core/data/dataTypes/Field.kt`. `Field` has a canonical `id` primary key and no `$id` requirement.
- Chat: `core/network/src/commonMain/kotlin/com/razumly/mvp/core/network/dto/ChatDtos.kt` declares canonical `id` first for both `ChatGroupApiDto` and `MessageApiDto`; both conversion functions choose `id` before `legacyId`.
- Billing and organizations: `core/repository-impl/src/commonMain/kotlin/com/razumly/mvp/core/data/repositories/BillingRepository.kt` declares canonical `id` first for `BillApiDto`, `BillPaymentApiDto`, `SubscriptionApiDto`, `ProductApiDto`, and `OrganizationApiDto`. Each conversion function chooses `id` before its optional fallback.
- Wear OS: `wearApp/src/main/java/com/razumly/mvp/wear/data/WearDtos.kt` declares canonical `id` for user, event, field, team, match, segment, and incident records. Every `resolvedId()` expression prefers `id`.
- watchOS: `iosApp/watchApp/WatchOfficialModels.swift` includes `id` in every relevant `CodingKeys` enum. User, event, field, team, match, segment, and incident `resolvedId` accessors prefer canonical `id`.

These decoders use Kotlin serialization or Swift `Codable`, both of which tolerate the legacy field being absent because the fallback properties are optional. Canonical-only responses therefore remain compatible with the complete v1.6.13 resource surface.

## Endpoint inventory and alias retirement

Exact-tag source and HTTP tests call `/api/billing/create_billing_intent` and `/api/billing/purchase-intent`. Neither v1.6.13 nor the current mobile source references the alternate billing spellings `billing_intent`, `billing-intent`, `purchase_intent`, or `create_purchase_intent`. Neither version references the user spellings `invite-email`, `invite-by-email`, `invite_by_email`, `exists`, `exists-by-email`, or `lookup-by-email`.

The web server therefore removes those ten App Router source files. `src/server/__tests__/retiredApiAliases.test.ts` prevents them from returning. The canonical billing endpoints remain unchanged. `/api/boldsign/webhook` remains because it is an external BoldSign callback path, not a mobile-client-version alias; deleting it requires provider configuration evidence outside this compatibility-floor inventory.

Reproduce the endpoint search from `/Users/elesesy/StudioProjects/mvp-app-critical-audit`:

    git grep -n -E 'api/billing/(create_billing_intent|purchase-intent)' v1.6.13 -- '*.kt' '*.swift'
    git grep -n -E 'billing_intent|billing-intent|purchase_intent|create_purchase_intent|invite-email|invite-by-email|invite_by_email|exists-by-email|lookup-by-email' v1.6.13 -- '*.kt' '*.swift'

The first command prints the two canonical billing paths. The second command prints no production call site for any retired alias.
